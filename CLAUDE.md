# CLAUDE.md — правила работы в проекте Square

## Язык общения (важнейшее правило)

Всё общение с пользователем — **на русском**: ответы, планы (содержимое plan-файлов в Plan Mode),
уточняющие вопросы, предложения по подходу. Это не зависит от языка кода (см. «Язык кода» ниже) —
код и комментарии всё равно только на английском.

## Принципы разработки

Придерживаемся **SOLID, DRY, KISS** и — в первую очередь — **YAGNI**.

- **YAGNI (главное).** Не пишем код «на всякий случай» / «потом пригодится». Реализуем только то,
  что нужно для текущей задачи прямо сейчас. Никаких спекулятивных абстракций, флагов, заголовков,
  слоёв и «заделов на будущее» без конкретного, уже существующего требования.
  - Не делаем псевдо-механизмов (например, «как бы security», который на деле ничего не защищает) —
    если настоящая потребность появится позже, тогда и сделаем под реальные условия.
- **KISS.** Самое простое решение, которое решает задачу. Не усложняем без причины.
  - Предпочитаем минимальное изменение, которое решает проблему, а не «заодно» рефакторим рядом
    лежащее.
  - Одно понятное правило лучше нескольких пересекающихся флагов/настроек, означающих почти одно
    и то же. Если поведение можно вывести из уже существующего состояния/поля — не добавляем для
    этого ещё один булевый флаг.
- **DRY.** Перед написанием нового ищем существующие функции/утилиты/паттерны и переиспользуем их.
- **SOLID.** Чистые интерфейсы и разделение ответственности — там, где это реально упрощает код,
  а не ради самих принципов (см. YAGNI/KISS).
  - **S.** Controller — тонкий слой (приём запроса, валидация, делегирование в `I*Service`), не
    смешиваем HTTP/DTO-логику с бизнес-правилами и SQL/EF-запросами.
  - **O.** Расширяем поведение новыми реализациями/данными (например, новый `dgr_type` со своими
    `extra_props`), а не очередным `switch`/`if` по строковому типу, разбросанным по контроллерам.
  - **L.** Любая реализация интерфейса (`I*Service`) должна соблюдать его контракт — нет «особых»
    реализаций, которым нужна дополнительная подготовка вызывающей стороны, которой не требует
    остальные.
  - **I.** Интерфейсы — узкие, под конкретного потребителя; не передаём «весь сервис», когда нужен
    один метод.
  - **D.** Зависимости — через конструктор (DI), как уже принято в проекте; не лезем за
    статическими синглтонами/глобальным состоянием там, где есть DI-контейнер.

При сомнении «добавить ли запас прочности» — по умолчанию **не добавляем** и спрашиваем.

## Язык кода

Переписка с пользователем — на русском. Код (идентификаторы, комментарии, commit-сообщения) —
**только на английском**, независимо от языка переписки с пользователем. Существующие русские
комментарии в коде не обязательно переписывать просто так, но новый код и новые комментарии —
английский.

## Формат кода

Старайтесь не превышать **100 символов** в строке. Если строка длиннее — переносите идиоматично
для языка (C#: разбивка аргументов/цепочек вызовов; TS/TSX: JSX-атрибуты или chained calls на
новых строках). Не жертвуйте читаемостью ради переноса любой ценой.

## Прежде чем менять то, что уже работает

Если пользователь говорит, что что-то **работает**, или в репозитории/конфигах виден
стабильный, осознанный setup (launch.json, порты, пути окружения, докер-компоновка) — не меняйте
этот подход в рамках той же задачи, если пользователь явно не просил именно это поменять. При
сомнении — коротко спросите, что и зачем меняете, прежде чем редактировать.

## Env-переменные и launch.json

При добавлении/изменении переменных окружения, которые читает бэкенд (`EnvConfig.Require`,
`IConfiguration`, и т.п.), синхронизируйте **`.vscode/launch.json`** — добавляйте те же
переменные в `env` каждой конфигурации, которая запускает соответствующий процесс, иначе F5
тихо разъезжается с docker-compose/продакшеном.

## Кодировка файлов

Перед сохранением любого изменённого файла проверяйте, что он в UTF-8 без BOM (`file -i <путь>`
должен показывать `charset=utf-8` или `us-ascii`; BOM — `file <путь>` не должен показывать
`with BOM`). Если файл оказался в другой кодировке (например, Windows-1251) — конвертируйте в
UTF-8 (`iconv -f <исходная> -t utf-8`), сохраняя содержимое. Если в файле уже встречаются байты
U+FFFD («битые» символы) — это не проблема кодировки самого файла, а уже случившаяся потеря
данных (исходные байты не восстановить простой конвертацией); в этом случае предупредите
пользователя явно, а не пытайтесь «починить» молча.

## Проверка после реализации

После завершения плана или многофайловой фичи прогоняйте сборку/линт затронутых частей —
`dotnet build LeafletAlarms.sln -c Debug` для бэкенда, `npm run build` и `npm run lint` для
`tree_alarm.client`. Не считайте задачу завершённой, если сборка или линт падают в тех модулях,
которые вы поменяли (если ошибка явно предсуществующая и не связана с вашим изменением — скажите
об этом явно, а не тихо пропускайте проверку).
